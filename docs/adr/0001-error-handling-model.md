# ADR-0001: 错误处理两层模型

## 元数据
- **状态**：已采纳
- **日期**：2026-06-21
- **决策者**：AI agent + 用户

## 上下文
项目由 Electron 主进程和渲染进程组成，通过 IPC 通信。代码中存在三类位置产生异常：
1. 纯函数（数据转换、文件系统查询）
2. 边界层（IPC handler、事件监听器、副作用 hook）
3. 全局未捕获异常

需要统一的错误处理策略，避免异常导致进程崩溃或 UI 白屏。

## 选项

- **选项 A：到处 try/catch** — 每个函数内部自己捕获并处理
  - 优：简单直接
  - 劣：错误处理逻辑散落，纯函数签名被污染，测试复杂度翻倍

- **选项 B：两层模型（采纳）** — 纯函数自然抛出，边界层统一捕获
  - 优：纯函数保持简洁可测试，边界层集中处理，职责清晰
  - 劣：需严格区分"纯函数"和"边界层"

- **选项 C：Result<T,E> 统一返回** — 所有函数返回 Result 类型
  - 优：类型安全，调用方强制处理错误
  - 劣：引入新依赖或自建类型，所有现有函数签名需改动，与 React 生态不匹配

## 决策
选择选项 B：两层模型。

- 纯函数（`src/main/files.ts`、`src/main/search.ts` 等）自然抛出，不写 try/catch
- 边界层（IPC handler、`ipcMain.on` 监听器、React 副作用 hook）统一捕获
- 全局 `process.on('uncaughtException')` / `'unhandledRejection'` 仅作最后防线日志，不依赖其恢复

## 后果
- **正面**：纯函数保持简洁，测试只需测正常路径 + 预期异常；边界层错误统一用 `logError` 记录
- **负面**：开发者需清楚判断自己写的是"纯函数"还是"边界层"；忘记在边界捕获会触发全局处理器
- **迁移**：已完成全部边界层包裹（IPC handler → index.ts、事件监听器 → window/menu/watcher、hook → useWorkspaceInit 等 6 处、组件 → ErrorBoundary）
