# ADR-0002: IPC 通信模式

## 元数据
- **状态**：已采纳
- **日期**：2026-06-21
- **决策者**：AI agent + 用户

## 上下文
Electron 提供多种 IPC 模式：`invoke/handle`（请求-响应）、`send/on`（单向推送）、`postMessage`（MessagePort）。项目需要统一规则，明确每种场景用哪种模式及如何处理异常。

## 选项

- **选项 A：全部用 invoke/handle** — 所有 IPC 通信都用请求-响应模式
  - 优：统一，调试方便
  - 劣：文件监控、菜单操作等单向推送场景不自然；调用方被迫等待不必要的响应

- **选项 B：混合模式按语义选择（采纳）** — 需要响应用的 `invoke/handle`，纯通知用 `send/on`
  - 优：语义匹配，不需要响应的地方不阻塞
  - 劣：两种模式错误处理规则不同，需文档明确

- **选项 C：全部用 send/on + 回调** — 所有通信用推送 + 独立回调
  - 优：最大灵活性
  - 劣：回调地狱，状态管理复杂

## 决策
选择选项 B：混合模式。

- `invoke/handle`：文件操作、store 读写、对话框、shell 打开
  - 错误处理：handle 内 catch → `logError` → rethrow，让渲染进程感知失败
- `send/on`：文件监控推送、搜索结果流式推送、菜单操作
  - 错误处理：on 内 catch → `logError`（即发即弃，无法通知发送方）
- `store:set` / `store:delete`：故意不 rethrow，静默失败 + 日志

## 后果
- **正面**：IPC 模式选择有据可依，新增通道时只需判断语义即可确定模式
- **负面**：需记住两种模式的不同错误行为；`send/on` 的发送方无法获知处理失败
- **迁移**：所有 8 个 IPC handler 已按此规则实现（`src/main/index.ts`）
